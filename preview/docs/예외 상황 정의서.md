# 예외 상황 정의서

## 1. 문서 목적

이 문서는 스냅챌 MVP에서 사용자가 예상과 다른 상황을 만났을 때 어떻게 안내하고 처리할지 정의한다.

MVP에서는 모든 예외를 완벽하게 처리하기보다, 핵심 플로우가 끊기지 않도록 최소한의 안내와 복구 방법을 제공하는 것을 목표로 한다.

## 2. 예외 처리 원칙

- 문구는 짧고 쉬운 한국어로 작성한다.
- 사용자가 다음에 무엇을 하면 되는지 바로 알 수 있어야 한다.
- MVP에서는 복잡한 운영 대응보다 재시도/재입력 중심으로 처리한다.
- 신고/차단/내보내기 같은 안전 기능은 MVP에서 제외한다.
- 회원가입이 없기 때문에 계정 복구/기기 변경 복구는 제공하지 않는다.
- 영상 내용 자체는 AI 분석하지 않는다.

## 4. 방 참여 관련 예외

## E-001. 참여 코드가 틀린 경우

### 발생 상황

사용자가 존재하지 않거나 잘못 입력한 참여 코드를 제출한다.

### 사용자 문구

```
방을 찾을 수 없어요
```

### 처리 방식

- 참여 코드 입력 화면에 그대로 머문다.
- 사용자는 코드를 다시 입력할 수 있다.

### 비고

- MVP에서는 코드 형식 제한을 따로 두지 않는다.

## E-002. 초대 링크가 잘못되었거나 만료된 경우

### 발생 상황

사용자가 유효하지 않은 초대 링크로 접근한다.

### 사용자 문구

```
방을 찾을 수 없어요
```

### 처리 방식

- 방 참여 실패 상태를 보여준다.
- 사용자는 참여 코드를 다시 입력하거나 새 링크를 받아야 한다.

### 비고

- 참여 URL은 바뀔 수 있으나 변경 조건은 미정이다.
- MVP에서는 링크 만료/재발급 정책을 깊게 다루지 않는다.

## E-003. 방 인원이 가득 찬 경우

### 발생 상황

방의 active 참여자가 최대 5명에 도달한 상태에서 추가 사용자가 참여하려고 한다.

### 사용자 문구

```
다 찬 방이에요
```

### 처리 방식

- 참여를 막는다.
- 참여 코드 입력 화면 또는 이전 화면으로 돌아가 다시 입력할 수 있게 한다.

### 정책

- 방 최대 인원은 5명이다.
- MVP 실제 테스트는 3명 기준이다.

## E-004. 모든 인원이 나가 방이 삭제된 경우

### 발생 상황

모든 사용자가 방을 나가서 방이 삭제된 뒤, 누군가 기존 링크/코드로 접근한다.

### 사용자 문구

```
방을 찾을 수 없어요
```

### 처리 방식

- 시작 화면 또는 참여 코드 입력 화면으로 이동한다.

### 정책

- 모든 인원이 나가면 방은 삭제된다.

## 5. 프로필/입력 관련 예외

## E-005. 프로필 이름이 비어 있는 경우

### 발생 상황

사용자가 프로필 이름 없이 참여하려고 한다.

### 사용자 문구

```
이름을 입력해 주세요
```

### 처리 방식

- 참여를 막고 입력 필드에 머문다.

### 정책

- 글자 수 제한은 MVP에서 두지 않는다.
- 단, 빈 값은 참여자 구분이 어려우므로 막는 것을 권장한다.

## E-006. 프로필 이름이 너무 긴 경우

### 발생 상황

사용자가 매우 긴 이름을 입력한다.

### 사용자 문구

- MVP에서는 제한하지 않으므로 별도 문구 없음

### 처리 방식

- MVP에서는 허용한다.
- UI 깨짐이 심하면 추후 제한을 둔다.

### 비고

- 고객 요청 기준으로 MVP에서는 이름 제한이 필요 없다고 결정했다.

## 6. 퀘스트 관련 예외

## E-007. 퀘스트 후보를 가져오지 못한 경우

### 발생 상황

사전 생성 퀘스트 데이터 조회에 실패하거나 후보 3개를 구성하지 못한다.

### 사용자 문구

- MVP에서는 별도 안내 없이 기본 후보를 보여준다.

### 처리 방식

- 기본 퀘스트 목록에서 후보를 대체한다.
- 기본 후보 예시:
    - 공통 주제 로그
    - 무지개 로그
    - 같은 하루 다른 시선

### 정책

- 사용자가 직접 퀘스트를 생성하지 않는다.
- 자동으로 대체한다.

## E-008. 오늘의 퀘스트가 이미 정해진 경우

### 발생 상황

첫 접근자가 이미 오늘의 퀘스트를 선택한 뒤 다른 사용자가 접근한다.

### 사용자 문구

- 별도 예외 문구 없음

### 처리 방식

- 이미 선택된 오늘의 퀘스트를 적용한다.
- 다른 사용자는 퀘스트 선택 화면을 보지 않는다.

### 정책

- 오늘의 퀘스트는 하루 동안 변경할 수 없다.

## E-009. 동시에 여러 명이 첫 접근한 경우

### 발생 상황

오늘의 퀘스트가 없는 상태에서 여러 사용자가 거의 동시에 첫 시간 로그에 접근한다.

### 사용자 문구

- 선택권을 얻지 못한 사용자는 별도 안내 없이 확정된 퀘스트 화면으로 이동

### 처리 방식

- 서버에서 먼저 퀘스트 선택 권한을 확정한 1명만 첫 접근자로 처리한다.
- 나머지 사용자는 확정된 퀘스트를 따른다.

### 정책

- 하루 첫 접근자는 1명만 존재해야 한다.

## 7. 촬영 관련 예외

## E-010. 촬영 중 앱이 꺼지거나 오류가 발생한 경우

### 발생 상황

카메라 화면에서 앱이 꺼지거나 촬영이 정상 완료되지 않는다.

### 사용자 문구

- 별도 문구 없이 촬영 화면부터 다시 시작

### 처리 방식

- 해당 시간대에 업로드된 영상이 없으면 다시 촬영 가능하다.
- 촬영부터 다시 시작하게 한다.

### 정책

- 촬영 실패는 영상 업로드로 간주하지 않는다.

## E-011. 카메라 권한이 거부된 경우

### 발생 상황

사용자가 카메라 권한을 허용하지 않는다.

### 사용자 문구

미정

### 처리 방식

미정

### 비고

- 권한 거부 처리 방식은 아직 결정되지 않았다.
- MVP 구현 전 결정 필요.

## E-012. 같은 시간대에 이미 촬영한 경우

### 발생 상황

사용자가 같은 시간대에 이미 영상을 업로드한 뒤 다시 촬영하려고 한다.

### 사용자 문구

```
이미 찍은 시간이에요
```

### 처리 방식

- 추가 촬영을 막는다.
- 기존 영상을 볼 수 있게 한다.

### 정책

- 같은 시간대에는 사용자당 1회만 가능하다.
- 단, 삭제한 경우 같은 시간대 안에서 재촬영 가능하다.

## E-013. 해당 시간대에 촬영하지 못한 경우

### 발생 상황

사용자가 촬영 가능한 시간대에 촬영하지 않는다.

### 사용자 문구

- 별도 안내 없음

### 처리 방식

- 해당 시간대는 실패/미참여 처리한다.
- 로그 화면에서는 빈 화면 또는 검은 화면으로 표시한다.
- 다음 시간대에는 다시 참여할 수 있다.

### 정책

- “못하겠어요” 버튼 없음
- “건너뛰기” 버튼 없음

## 8. 업로드/네트워크 관련 예외

## E-014. 영상 업로드 실패

### 발생 상황

촬영은 완료되었지만 서버 업로드에 실패한다.

### 사용자 문구

```
업로드에 실패했어요
```

### 처리 방식

- 팝업으로 실패를 알린다.
- 다시 업로드 버튼을 제공한다.
- 사용자가 다시 업로드를 누르면 같은 영상을 재업로드한다.

### 정책

- 업로드 실패를 조용히 넘기지 않는다.
- 사용자가 실패 상태를 알 수 있어야 한다.

## E-015. 로컬 서버에 연결할 수 없는 경우

### 발생 상황

MVP 시연 중 Android 앱이 로컬 서버에 접근하지 못한다.

### 사용자 문구

```
서버에 연결할 수 없어요
```

### 처리 방식

- 재시도 버튼을 제공한다.
- 같은 Wi-Fi 연결 또는 임시 터널링 상태를 확인한다.

### 비고

- MVP는 로컬 서버를 사용할 수 있다.
- 같은 네트워크 또는 임시 터널링이 필요하다.

## E-016. 저장/공유 실패

### 발생 상황

하루 로그 저장 또는 외부 공유가 실패한다.

### 사용자 문구

```
저장에 실패했어요
```

또는

```
공유에 실패했어요
```

### 처리 방식

- 사용자가 다시 시도할 수 있게 한다.

### 정책

- 저장/공유는 MVP 성공 기준이므로 실패 상태를 안내해야 한다.

## 9. 영상 삭제 관련 예외

## E-017. 다른 사람의 영상을 삭제하려는 경우

### 발생 상황

사용자가 본인이 올리지 않은 영상을 삭제하려고 한다.

### 사용자 문구

- 삭제 버튼을 노출하지 않는 방식 권장

### 처리 방식

- 다른 사람의 영상에는 삭제 기능을 제공하지 않는다.

### 정책

- 내가 오늘 올린 영상만 삭제 가능하다.

## E-018. 오늘이 아닌 과거 영상을 삭제하려는 경우

### 발생 상황

사용자가 과거 날짜의 영상을 삭제하려고 한다.

### 사용자 문구

- 삭제 버튼을 노출하지 않는 방식 권장

### 처리 방식

- MVP에서는 내가 오늘 올린 영상만 삭제 가능하다.

### 정책

- 과거 로그 안정화는 후순위다.

## E-019. 삭제 후 같은 시간대 재촬영 가능 여부

### 발생 상황

사용자가 오늘 올린 영상을 삭제한 뒤 같은 시간대 안에서 다시 촬영하려고 한다.

### 사용자 문구

- 별도 안내 없음

### 처리 방식

- 삭제한 경우 같은 시간대 안에서 재촬영 가능하다.

### 정책

- 삭제는 원본 완전 삭제가 아니라 공유 로그에서 삭제/숨김 처리다.

## 10. 방 나가기 관련 예외

## E-020. 사용자가 방을 나가는 경우

### 발생 상황

사용자가 설정 화면에서 방 나가기를 선택한다.

### 사용자 문구

```
방에서 나갈까요?
```

### 처리 방식

- 사용자를 참여자 목록에서 제거한다.
- 해당 사용자의 공유 로그를 삭제/숨김 처리한다.
- 방 안에서 해당 사용자의 칸 자체를 제거한다.

### 정책

- 방 나가기는 가능하다.
- 모든 인원이 나가면 방을 삭제한다.

## E-021. 모든 인원이 방을 나간 경우

### 발생 상황

방의 active 참여자가 0명이 된다.

### 사용자 문구

- 별도 사용자 문구 없음

### 처리 방식

- 방을 삭제 처리한다.

### 정책

- 방 삭제는 모든 인원이 나갔을 때 발생한다.

## 11. 앱 삭제/기기 변경 관련 예외

## E-022. 사용자가 앱을 삭제한 경우

### 발생 상황

사용자가 앱을 삭제했다가 다시 설치한다.

### 사용자 문구

- 별도 복구 안내 없음

### 처리 방식

- MVP에서는 기존 방/영상 접근 복구를 제공하지 않는다.
- 사용자는 새로 참여해야 한다.

### 정책

- 회원가입이 없기 때문에 복구하지 않는다.
- 회원가입은 고도화 전략이다.

## E-023. 사용자가 기기를 바꾼 경우

### 발생 상황

사용자가 다른 기기에서 기존 방에 접근하려고 한다.

### 사용자 문구

- 별도 복구 안내 없음

### 처리 방식

- MVP에서는 기존 사용자로 복구하지 않는다.
- 새 참여자로 들어가야 한다.

## 12. 악용/안전 관련 예외

## E-024. 원치 않는 사람이 방에 들어온 경우

### 발생 상황

초대 링크가 외부에 공유되어 원치 않는 사용자가 방에 들어온다.

### 사용자 문구

- MVP에서는 별도 안내 없음

### 처리 방식

- 신고/차단/내보내기는 MVP에서 제공하지 않는다.
- 사용자는 방 나가기를 할 수 있다.

### 정책

- 참여 코드는 바뀌지 않는다.
- 참여 URL은 바뀔 수 있다.
- 초대 URL 재발급 정책은 고도화 항목이다.

## E-025. 부적절한 영상이 올라온 경우

### 발생 상황

참여자가 원치 않는 영상이나 부적절한 영상을 올린다.

### 사용자 문구

- MVP에서는 별도 신고 문구 없음

### 처리 방식

- 신고/검수/차단은 MVP에서 제공하지 않는다.
- 사용자는 본인이 올린 영상만 삭제할 수 있다.

### 정책

- MVP에서는 깊게 고려하지 않는다.
- 안전 기능은 고도화 항목이다.

## 13. 빈 상태 처리

## 방에 아직 나 혼자인 경우

### 사용자 문구 예시

```
친구를 초대해 같이 시작해요
```

### 처리 방식

- 초대 링크/참여 코드 공유를 유도한다.

## 오늘 아직 아무도 찍지 않은 경우

### 사용자 문구 예시

```
아직 올라온 로그가 없어요
```

### 처리 방식

- 촬영 가능한 경우 촬영 진입을 제공한다.

## 특정 시간대에 아무도 찍지 않은 경우

### 사용자 문구 예시

- 별도 문구 없이 빈/검은 화면 중심 표시

### 처리 방식

- 셋로그와 유사한 표현을 사용한다.

## 14. MVP에서 미정인 예외

| 항목 | 왜 중요한가 |
| --- | --- |
| 카메라 권한 거부 처리 | 촬영 기능을 쓸 수 없기 때문에 핵심 플로우가 막힘 |
| 저장소 권한 거부 처리 | 저장/공유 기능에 영향 |
| 초대 URL 변경 조건 | 원치 않는 유입 방지와 관련 |
| 앱 미설치 상태에서 초대 링크 클릭 | APK 시연 환경에서는 깊게 처리하지 않을 수 있음 |
| 과거 로그 삭제/수정 | 과거 로그 안정화가 후순위이기 때문 |

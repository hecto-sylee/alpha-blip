# 핵심 사용자 시나리오

## 1. 문서 목적

이 문서는 스냅챌 MVP에서 사용자가 실제로 어떤 흐름으로 서비스를 사용하는지 정리한 문서다.

기능 목록이 아니라, 사용자의 입장에서 “처음 앱을 켜는 순간부터 하루 로그를 저장/공유하는 순간까지”의 흐름을 설명한다.

## 2. 주요 사용자 역할

### 방 생성자

- 처음 방을 만드는 사용자
- 방 이름, 모드, 프로필 이름을 입력함
- 초대 링크 또는 참여 코드를 다른 사람에게 공유함

### 참여자

- 초대 링크 또는 참여 코드로 방에 들어오는 사용자
- 프로필 이름을 입력하고 바로 참여함
- 별도 회원가입은 하지 않음

### 하루 첫 접근자

- 해당 날짜의 첫 번째 시간 로그에 가장 먼저 접근한 사용자
- 그날의 퀘스트 주제를 선택함
- 방장과는 다름. 매일 바뀔 수 있음

### 일반 참여자

- 이미 정해진 오늘의 퀘스트를 확인하고 촬영하는 사용자
- 퀘스트를 바꾸지는 않음

## 3. 전체 핵심 흐름 요약

1. 방 생성자가 방을 만든다.
2. 방 생성자가 초대 링크 또는 참여 코드를 공유한다.
3. 참여자가 링크/코드로 방에 들어온다.
4. 하루 첫 접근자가 오늘의 퀘스트 후보 3개 중 1개를 선택한다.
5. 오늘의 퀘스트가 하루 동안 고정된다.
6. 각 사용자는 시간대별로 촬영 직전 세부 미션을 확인한다.
7. 사용자는 2초 영상을 촬영한다.
8. 촬영된 영상이 시간대별 로그에 쌓인다.
9. 사용자는 친구/커플 영상을 보고 이모지 반응을 남긴다.
10. 하루 끝에 로그를 저장/공유한다.

## 4. 첫 사용 시나리오: 방 만들기

### 상황

사용자가 친구 또는 커플과 함께 스냅챌을 시작하려고 한다.

### 사용자 흐름

1. 사용자가 앱을 실행한다.
2. 사용자가 새 방 만들기를 선택한다.
3. 사용자가 방 이름을 입력한다.
4. 사용자가 모드를 선택한다.
    - 친구 모드
    - 커플 모드
5. 사용자가 자신의 프로필 이름을 입력한다.
6. 앱이 방을 생성한다.
7. 앱이 참여 코드와 초대 링크를 생성한다.
8. 사용자가 초대 링크 또는 참여 코드를 다른 사람에게 공유한다.

### 기대 결과

- 방이 생성된다.
- 방 생성자는 해당 방의 첫 참여자로 등록된다.
- 다른 사용자가 링크 또는 코드로 참여할 수 있다.

### 예외

- 방 이름 또는 프로필 이름 제한은 MVP에서는 두지 않는다.
- 방 생성 실패 시 다시 시도하도록 안내한다.

## 5. 첫 사용 시나리오: 방 참여하기

### 상황

사용자가 친구 또는 커플에게 받은 링크/코드로 방에 들어간다.

### 사용자 흐름 A: 초대 링크 참여

1. 사용자가 초대 링크를 누른다.
2. 앱이 해당 방 정보를 확인한다.
3. 사용자가 프로필 이름을 입력한다.
4. 사용자가 방에 참여한다.

### 사용자 흐름 B: 참여 코드 입력

1. 사용자가 앱에서 참여 코드 입력을 선택한다.
2. 사용자가 참여 코드를 입력한다.
3. 앱이 해당 방 정보를 확인한다.
4. 사용자가 프로필 이름을 입력한다.
5. 사용자가 방에 참여한다.

### 기대 결과

- 사용자는 별도 회원가입 없이 방에 들어간다.
- 방 참여자는 참여자 목록에 추가된다.

### 예외

- 잘못된 코드 또는 링크: “방을 찾을 수 없어요” 표시 후 다시 입력
- 방 정원 초과: “다 찬 방이에요” 표시 후 다시 입력

## 6. 하루 시작 시나리오: 첫 접근자의 퀘스트 선택

### 상황

새로운 날짜가 시작되었고, 아직 오늘의 퀘스트가 정해지지 않았다.

### 사용자 흐름

1. 사용자가 해당 날짜의 첫 번째 시간 로그에 접근한다.
2. 앱이 오늘의 퀘스트가 아직 정해지지 않았는지 확인한다.
3. 가장 먼저 접근한 사용자를 하루 첫 접근자로 판단한다.
4. 앱이 추천 퀘스트 후보 3개를 보여준다.
5. 사용자가 후보 중 하나를 선택한다.
6. 선택한 퀘스트가 오늘의 로그 주제로 확정된다.
7. 오늘의 퀘스트는 하루 동안 변경할 수 없다.
8. 사용자는 이후 일반 촬영 흐름으로 이동한다.

### 화면에 보여야 하는 정보

- 추천 주제명
- 다른 주제 선택 버튼 또는 후보 선택 영역
- 선택 후 촬영으로 넘어가는 흐름

### 기대 결과

- 오늘의 퀘스트 주제가 확정된다.
- 모든 참여자가 같은 주제 기준으로 하루 로그를 진행한다.

### 예외

- 퀘스트 후보 생성 실패 시 기본 퀘스트 목록에서 대체한다.

## 7. 일반 촬영 시나리오

### 상황

오늘의 퀘스트가 이미 정해져 있고, 사용자가 특정 시간대에 촬영하려고 한다.

### 사용자 흐름

1. 사용자가 오늘의 로그 화면에 접근한다.
2. 사용자가 현재 시간대 또는 촬영 가능한 시간대에 진입한다.
3. 앱이 촬영 직전 세부 미션을 보여준다.
4. 세부 미션은 해당 사용자에게만 보인다.
5. 사용자가 미션 안내를 터치한다.
6. 앱이 촬영 화면으로 이동한다.
7. 사용자가 2초 영상을 촬영한다.
8. 앱이 영상을 업로드한다.
9. 업로드가 완료되면 해당 시간대 로그에 영상이 표시된다.

### 기대 결과

- 사용자는 해당 시간대에 1개의 영상을 올린다.
- 영상은 촬영자, 날짜, 시간대, 주제, 세부 미션과 함께 저장된다.

### 예외

- 촬영 중 앱 오류: 촬영부터 다시 시작
- 업로드 실패: 팝업 표시 후 다시 업로드 버튼 제공
- 이미 해당 시간대에 영상을 올린 경우: 추가 촬영 불가
- 단, 사용자가 영상을 삭제한 경우 같은 시간대 안에서 재촬영 가능

## 8. 미참여 시나리오

### 상황

사용자가 특정 시간대에 촬영하지 못했다.

### 사용자 흐름

1. 사용자가 해당 시간대에 촬영하지 않는다.
2. 별도의 건너뛰기 버튼은 누르지 않는다.
3. 해당 시간대는 실패 또는 미참여로 남는다.
4. 사용자는 다음 시간대에 다시 참여할 수 있다.

### 기대 결과

- 미참여 시간대는 셋로그처럼 빈 화면 또는 검은 화면으로 표시된다.
- 사용자는 다음 시간대 참여에 제한을 받지 않는다.

### 정책

- “못하겠어요” 또는 “건너뛰기” 기능은 MVP에 없다.
- 촬영하지 못하면 그 시간대는 그냥 미참여로 처리한다.

## 9. 로그 보기 시나리오

### 상황

사용자가 친구/커플이 올린 영상을 확인하려고 한다.

### 사용자 흐름

1. 사용자가 로그 화면에 진입한다.
2. 사용자가 시간대별 로그를 확인한다.
3. 특정 시간대를 선택한다.
4. 해당 시간대에 참여한 사용자들의 영상을 본다.
5. 참여하지 않은 사용자는 빈 화면 또는 검은 화면으로 표시된다.
6. 사용자가 마음에 드는 영상에 이모지 반응을 남긴다.

### 기대 결과

- 사용자는 하루가 어떻게 쌓이고 있는지 확인한다.
- 친구/커플의 참여 여부를 볼 수 있다.
- 이모지로 가볍게 반응할 수 있다.

## 10. 저장/공유 시나리오

### 상황

하루 로그가 쌓였고, 사용자가 결과물을 저장하거나 SNS에 공유하려고 한다.

### 사용자 흐름

1. 사용자가 하루 로그를 확인한다.
2. 사용자가 저장 또는 공유를 선택한다.
3. 앱이 셋로그와 유사한 형태의 공유 결과물을 제공한다.
4. 사용자가 기기에 저장하거나 외부 SNS로 공유한다.

### 기대 결과

- 하루의 퀘스트 결과물이 추억처럼 저장된다.
- 사용자는 외부 SNS에 공유하고 싶어지는 결과물을 얻는다.

### 참고

- 워터마크는 MVP에서는 제외한다.
- 워터마크는 향후 고도화에서 검토한다.

## 11. 영상 삭제 시나리오

### 상황

사용자가 오늘 올린 영상을 로그에서 지우고 싶다.

### 사용자 흐름

1. 사용자가 자신이 오늘 올린 영상을 선택한다.
2. 사용자가 삭제를 선택한다.
3. 앱이 공유되는 로그에서 해당 영상을 삭제/숨김 처리한다.
4. 같은 시간대 안이라면 사용자는 다시 촬영할 수 있다.

### 기대 결과

- 공유 로그에서는 해당 영상이 보이지 않는다.
- 원본 파일 완전 삭제가 아니라 공유 로그에서만 삭제/숨김 처리한다.

### 정책

- 사용자는 내가 오늘 올린 영상만 삭제할 수 있다.
- 다른 사용자의 영상은 삭제할 수 없다.

## 12. 방 나가기 시나리오

### 상황

사용자가 더 이상 해당 방에 참여하지 않으려고 한다.

### 사용자 흐름

1. 사용자가 설정 페이지에 들어간다.
2. 사용자가 방 나가기를 선택한다.
3. 앱이 사용자를 방에서 제거한다.
4. 해당 사용자의 공유 로그는 삭제/숨김 처리된다.
5. 방 안에서 그 사람의 칸 자체를 없앤다.
6. 모든 인원이 방을 나가면 방을 삭제한다.

### 기대 결과

- 남은 사용자는 나간 사용자의 칸을 보지 않는다.
- 모든 인원이 나간 방은 유지하지 않는다.

## 13. 설정 시나리오

### 상황

사용자가 자신과 관련된 설정을 확인하거나 변경하려고 한다.

### 사용자 흐름

1. 사용자가 설정 페이지에 진입한다.
2. 사용자가 자신과 관련된 정보를 확인한다.
3. 필요 시 방 나가기를 수행한다.
4. MVP 편의를 위해 숨겨진 퀘스트 관리 기능에 접근할 수 있다.

### 기대 결과

- 별도 운영자용 관리자 페이지 없이 사용자용 설정 페이지에서 최소 관리가 가능하다.
- 퀘스트 관리는 MVP 편의를 위해 숨겨진 형태로 둔다.

## 14. 반복 사용 루프

1. 오늘의 퀘스트가 정해진다.
2. 사용자는 시간대별 세부 미션을 확인한다.
3. 사용자는 2초 영상을 촬영한다.
4. 친구/커플의 영상을 본다.
5. 이모지 반응을 남긴다.
6. 하루 로그가 쌓인다.
7. 하루 끝에 저장/공유한다.
8. 다음 날 새로운 퀘스트로 다시 시작한다.

## 15. 이탈 가능성이 높은 지점

- 퀘스트가 재미없음
- 촬영이 귀찮음
- 촬영할 상황이 아님
- 친구가 안 찍음
- 하루 결과물이 기대보다 재미없음
- 링크/코드 참여가 잘 안 됨
- 업로드 실패가 반복됨

## 16. 시연용 대표 시나리오

### 알파팀 시연 흐름

1. A가 방을 만든다.
2. A가 친구 모드를 선택한다.
3. A가 초대 링크를 B, C에게 공유한다.
4. B, C가 링크로 들어와 프로필 이름을 입력한다.
5. A가 하루 첫 접근자로 퀘스트 후보 3개를 본다.
6. A가 “무지개 로그” 같은 퀘스트를 선택한다.
7. A, B, C가 각자 촬영 직전 세부 미션을 확인한다.
8. 세 명이 2초 영상을 촬영한다.
9. 로그 화면에서 세 명의 영상이 같은 시간대에 표시된다.
10. B가 A의 영상에 이모지 반응을 남긴다.
11. A가 하루 로그를 저장/공유한다.

### 시연에서 보여줘야 하는 포인트

- 셋로그형 흐름을 유지한다.
- 하루 첫 접근자가 퀘스트를 선택한다.
- 퀘스트가 하루의 촬영 이유를 만든다.
- 친구/커플 모드에 따라 퀘스트가 달라질 수 있다.
- 결과물이 저장/공유 가능한 추억으로 남는다.

# MVP 기획 개요

> MVP 구현 범위(**S1 산책 매칭 + S3 기록 + 공통**)에 대한 상세 기획서다.

---

## MVP 한 줄 정의

> 산책 중인 강아지를 실시간으로 연결하고, 산책이 끝나면 기록을 남기게 하는 모바일 웹 서비스.

---

## 서비스명 — blip

> **blip** — 지도 위에 *지금 근처에 뜬 산책 친구*(레이더의 blip)와, *2초짜리 짧은 순간 기록*(a blip)을 한 단어에 담았다.

| 이유 | 설명 |
|---|---|
| 두 축을 한 단어에 | 레이더 *blip* = S1 실시간 매칭(획득 훅) / 짧은 순간 *blip* = S3 2초 클립 기록(리텐션) |
| 글로벌·Gen-Z | 짧은 소문자 한 단어. 숏폼 소셜캠(BeReal·Locket 계열) 작명 톤 |
| 종(種) 무관 | 강아지에 묶이지 않는 이름 → MVP는 강아지에 집중하되, 이후 반려동물 확장을 막지 않는다 |

---

## MVP 목표

MVP는 두 가지 핵심 가설을 검증한다.

| # | 가설 | 검증 질문 |
|---|---|---|
| 가설 1 | 사람들은 근처 산책 중인 강아지와 즉흥적으로 매칭해 함께 산책한다 | 실시간 매칭이 실제로 성사되는가? |
| 가설 2 | 산책 후 기록을 남기는 경험이 재방문·루틴 형성으로 이어진다 | 매칭이 없어도 기록을 위해 앱을 켜는가? |

**S1(매칭)** 은 앱으로 사람을 끌어오는 훅, **S3(기록)** 은 사람을 머무르게 하는 루틴.
두 레이어가 함께 작동하는지를 보는 것이 MVP의 핵심이다.

기록은 **2초 영상 클립**을 기본 단위로 하고, **산책 퀘스트(F-12)** 가 "오늘 뭘 기록하지?"라는 빈 페이지 문제를 풀어 매일 기록할 이유를 만든다. 즉 퀘스트는 가설 2(기록 루틴화)를 직접 겨냥한 장치다.

## 구현 범위

| 시스템 | 기능 |
|---|---|
| 공통 | F-02 반려동물 프로필 / F-09 개인정보 보호 설정 |
| S1 산책 매칭 | F-01 실시간 산책 지도 / F-03 같이 산책하기 요청 / F-04 매칭 세션 / F-05 매칭 로그 |
| S3 기록 | F-10 산책 기록 다이어리(2초 클립) / F-11 방(Room) 기반 공유 기록 / F-12 산책 퀘스트 |

## 타깃 사용자

**1차 타깃:** 도심·주거 밀집 지역에서 강아지를 산책시키는 보호자

- 매일 또는 주기적으로 산책한다
- 산책 중 다른 강아지와 자연스럽게 만나고 싶다
- 산책 기록을 남기고 쌓는 데 흥미가 있다

> MVP는 강아지 보호자에 집중한다.

## 제품 형태

| 항목 | MVP |
|---|---|
| 형태 | 모바일 웹 (반응형) |
| 기준 환경 | 모바일 브라우저 (산책 중 사용) |
| 위치 수집 | Browser Geolocation API (`watchPosition`) |
| MVP 구현 방식 | FastAPI 단일 서버 + 웹 SPA + Android WebView + ngrok 시연 (상세: [10_dev_stack.md](./10_dev_stack.md)) |

> 웹의 한계: 백그라운드 위치 추적이 약하다. MVP는 **앱 실행(포그라운드) 중 산책**을 전제로 한다.

## 핵심 검증 지표

### S1 산책 매칭

| 지표 | 의미 |
|---|---|
| 산책 시작 클릭률 | 핵심 기능 진입 |
| 위치 공유 허용률 | 사용 가능 사용자 비율 |
| 산책 요청률 / 수락률 | 핵심 행동 전환 + 매칭 품질 |
| 매칭 성공률 | 전체 전환 |

### S3 기록

| 지표 | 의미 |
|---|---|
| 산책 후 기록 작성률 | 루틴 형성 핵심 |
| 퀘스트 선택률 · 완주율 | 퀘스트가 기록 이유를 만드는가 |
| 혼자 산책 기록 비율 | 매칭 비종속 사용 여부 |
| 방 생성·업로드율 | 공유 레이어 작동 여부 |
| 재방문율 | 지속 사용 여부 |
